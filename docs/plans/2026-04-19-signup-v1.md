# OpenSignup v1 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship a self-hostable, ad-free, open-source coordination tool (sign-up coordination tool) to a first real user — a youth basketball team — with a foundation sufficient to scale through v3 without a schema rewrite.

**Architecture:** Next.js 15 App Router monolith (TypeScript) backed by Postgres via Drizzle, Auth.js for organizer magic-link auth, Zod as the single source of truth for types/validators/OpenAPI, pg-boss for Postgres-backed scheduled jobs, React Email templates rendered through a pluggable transport (console/SMTP/Resend). Hexagonal ports/adapters for every external integration so the identical code paths run hosted or self-hosted with only environment variables changing.

**Tech Stack:** Node 20+, Next.js 15 (App Router), TypeScript (strict), Postgres 15+, Drizzle ORM + drizzle-kit, Zod v3, Auth.js v5, pg-boss, React Email, pino, Vitest + Playwright, Tailwind + shadcn/ui, pnpm, Docker/Docker Compose.

---

## Build Sequencing Rationale

Twelve phases, in strict dependency order. Every phase ends on a green build and a commit. The foundation (Phase 0–2) is heavier up-front than strictly necessary for v1 because the PRD's fourth principle is "the advanced layer goes in first" — workspace scoping, stable refs, activity events, and pluggable adapters are cheap now and painful to retrofit.

The critical path to "basketball team uses it" is: Phase 0 → 1 → 2 → 3 → (subset of 4, 5, 6) → 9 → 10. Phases 7, 8, 11, 12 harden the product for the public beta. Phase 8 (organizer UI) must ship before the first user, but can reuse shadcn primitives aggressively to stay tight.

## Non-Negotiables

1. **TDD for pure logic.** Capacity enforcement, slug generation, ID encoding, email typo suggestion, authz policy — all tests-first. UI is not TDD'd; it's built with Playwright smokes at the end.
2. **Zod is the source of truth.** Every endpoint has one request schema and one response schema. Types are inferred. OpenAPI is generated.
3. **Every mutation goes through a pure function** that accepts `{ actor, workspace, input }` and returns `Result<Output, ServiceError>`. The route handler is a thin adapter. The future MCP server wraps the same pure function.
4. **Workspace scoping is enforced at the policy layer.** No route queries the database directly without going through the policy module. Every tenant table query has `workspace_id = ?` in the WHERE clause.
5. **No vendor lock-in.** Any dependency that requires an account (Resend, Sentry, PostHog) must be opt-in via env var and have a console or noop default.
6. **Commit after every green test suite.** Rollback must always be a single `git reset`.

## Phase 6 — Commitments (Participant Flow, Server Side)

### Task 6.1: `POST /api/slots/{id}/commitments` — the hot path

**This is the single most important service function.** Participants under load; capacity race must never double-book.

**Algorithm:**
1. Begin transaction.
2. Upsert Participant by `(signup_id, normalized_email)` — if exists and name mismatches, take the latest name.
3. Compute next `position = count(*) where slot_id = ?` (within transaction using `FOR UPDATE` on the slot row).
4. If slot has capacity and `position >= capacity`: rollback, return `capacity_full` with alternatives.
5. Insert commitment with `(slot_id, position)`; the unique constraint is the final safety net.
6. Emit `commitment.created` activity.
7. Schedule reminder job (pg-boss) for `slot_date - 48h` unless in past.
8. Commit transaction.
9. Return commitment with edit token (signed HMAC of commitment id) for the response only; we store only its hash.

**Tests (critical):**
- 100 concurrent commits to a 50-capacity slot → exactly 50 succeed.
- `capacity_full` returns next 3 open slots in same signup as alternatives (per PRD §8 drawer behavior).
- Duplicate email in same signup updates the participant row, does not create a new one; if a commitment for the same participant already exists on the same slot, return `conflict` with the existing commitment.
- Signup `status != 'open'` → `closed` error.
- Rate-limited by IP after 10/min on a given signup.

**Commit:** `feat(commitments): capacity-safe commit service`.

### Task 6.2: `GET /api/commitments/{id}?token=...`

Validates signed token matches stored hash. Returns participant's own commitment view (editable shape).

### Task 6.3: `PATCH /api/commitments/{id}?token=...`

Participant can edit notes, name, swap slot (which is: cancel old + create new atomically). Swap re-enforces capacity.

### Task 6.4: `DELETE /api/commitments/{id}?token=...`

Sets `status='cancelled'`, `cancelled_at=now()`. Frees the position slot (relax: positions don't need to be contiguous; capacity check counts `status in ('confirmed','tentative')`). Cancels the pg-boss reminder job if scheduled.

## Phase 10 — Reminders & Jobs

### Task 10.1: pg-boss bootstrap

**Files:** `src/jobs/boss.ts`, `src/jobs/runner.ts`, `scripts/worker.ts`

Singleton pg-boss instance using `DATABASE_URL`. Schema: `pgboss`. Queue names: `email.reminder`. Worker script starts pg-boss and subscribes.

Next.js server does not run the worker. `pnpm worker` runs it separately. Hosted instance runs it as a Fly process; self-hosters run `node dist/worker.js`. Dev: `pnpm dev` + `pnpm worker` in two terminals; Procfile or Foreman for convenience.

### Task 10.2: Schedule reminder on commit

Inside the commit transaction, publish a `email.reminder` job with `startAfter: slot_date - 48h`, payload `{ commitmentId }`. Use `singletonKey: commitmentId` to allow deduplication on swap/edit.

If the slot is within 48h, skip scheduling.

### Task 10.3: Reminder worker

Handler:
1. Load commitment + slot + signup (fail if orphaned or cancelled: noop).
2. Render `reminder.tsx` with context.
3. Send via email transport.
4. Record `reminder.sent` activity.

Retries: pg-boss default (3 attempts, exponential backoff). Dead-letter queue logs to Sentry if configured.

### Task 10.4: Tests

- Scheduling: on commit, assert a job row exists with correct `startAfter`.
- Cancel: on delete, assert job cancelled.
- Swap: on edit to different slot, assert old job cancelled + new job scheduled.
