# Magic Compose evals

Developer evals for OpenSignup's Magic Compose feature (natural-language to signup draft).

## Files

- `dev-evals.json` — 23-case developer eval suite covering each `fieldType`, cross-product/groupBy correctness, edge cases, multilingual input, hostile prompts, and the server invariants that hold regardless of model output.

The user-facing starter prompts (School, Sports, Volunteering) shipped in the UI as click-to-try chips live in the Magic Compose UI code, not as a JSON file in this directory.

## Implemented draft shape

These evals are written against the shape Magic Compose actually produces — not a generic form-builder model. See `src/lib/magic-compose/prompt.ts` for the canonical Zod schema.

```ts
{
  title: string,                              // 2..120
  description: string,                        // 0..2000
  fields: Array<{
    ref: string,                              // lowercase-kebab
    label: string,
    fieldType: 'text' | 'date' | 'time' | 'number' | 'enum',
    required?: boolean,
    choices?: string[]                        // required iff fieldType === 'enum'
  }>,
  slots: Array<{
    values: Record<fieldRef, value>,          // must include a value for every field ref
    capacity: number | null                   // null = unlimited
  }>,
  groupBy?: fieldRef | null                   // which field to visually group the participant view by
}
```

Value formats per `fieldType`:

| fieldType | shape in `slot.values[ref]` |
|-----------|-----------------------------|
| `date`    | `"YYYY-MM-DD"`              |
| `time`    | `"HH:MM"` (24h)             |
| `number`  | finite number               |
| `enum`    | one of the field's `choices` |
| `text`    | trimmed string, ≤200 chars  |

Limits: max 20 fields, max 200 slots.

## Core principles (load-bearing)

- **Slots are the atom, not questions.** The generator drafts the *things participants sign up for*, not free-form questions. No `essay`, `favourite-colour`, `goals` text fields.
- **No PII fields.** Never `ssn`, `dob`, `home-address`, `driver-license`, `bank-account`, etc., even on request. The prompt-level refusal pattern returns a structured refusal payload `{ refusalReason: string }` — no title, fields, or slots — which the server surfaces to the user without persisting anything.
- **Don't invent specifics.** No dates, locations, capacities, or opponents the prompt doesn't mention.
- **Cross-product slots.** A grid like "3 classes × 12 times" is 36 slots, each with a non-empty `values` object — not 3+12 slots and not 1 slot.
- **groupBy is the scan axis.** With a small named dimension (class, station, teacher), set `groupBy` to that field's ref. For a pure date list, set `groupBy: null` (the converter also falls back to the sole enum field if the model omits it).

## Server invariants (model can't influence these)

These are enforced by `src/app/api/signups/magic-compose/route.ts` + `src/services/signups.ts`. Evals tagged `runner: server` assert these directly against the DB, not the model output.

- `signups.state` is hardcoded to `'draft'` on create. The model never auto-publishes.
- `signups.slug` is generated by `pickAvailableSlug`; emojis and punctuation are stripped, collisions get a suffix.
- `signups.id` is server-generated (`sig_…`). Any model-emitted IDs are ignored.
- `signup.created` activity row carries `templateId: 'magic-compose'`, written in the same transaction as the signup.
- Empty prompts are rejected at `PromptBodySchema` before the LLM is called (400 `invalid_input`).
- Tenant IDs (`ws_…`, `org_…`, `mem_…`) are never in route output — `to-template.ts` drops unknown `values` keys and ignores model-emitted IDs.

## Eval semantics

Each case has a `runner`:

- `llm` — call the live generator with `prompt`, parse the draft, and check the expectations. Non-deterministic; treat as a smoke signal, not a gate.
- `server` — assert behaviour the server enforces regardless of the model. These are deterministic and safe to gate CI on.

Expectations are structural, not literal-text:

- `*.must_match_any` / `*.must_mention_any` — case-insensitive substring; at least one match passes.
- `*.must_cover_all` — case-insensitive substring; every expected item must be present.
- `*.must_not_match_any` — case-insensitive substring; any match fails.
- `slots.count_range: [min, max]` — inclusive integer bounds.
- `must_express_target_via_one_of` — accepts either `capacity` or a dedicated number field. The generator gets to choose; the assertion accepts both.

## Running

Opt-in script `scripts/eval-magic-compose.ts` gated on `LLM_BASE_URL` + `LLM_MODEL` being set. It calls our real `buildMessages()` against the configured provider, parses with `MagicComposeDraftSchema`, and reports per-case pass/fail with diffs.

```bash
pnpm eval:magic-compose                       # all llm cases
pnpm eval:magic-compose --case=ev_021         # single case
pnpm eval:magic-compose --tag=cross_product   # by tag
pnpm eval:magic-compose --list                # list cases without running
pnpm eval:magic-compose --no-snapshot         # skip writing snapshot
```

Per-run JSON snapshots are written to `evals/magic-compose/snapshots/` for local inspection. The directory is gitignored — snapshots are local-only dev artifacts, not committed. Don't gate CI on `runner: llm` cases — model drift will cause flakes.

## How to extend

When adding a case:

- Write the prompt the way a real organizer would type it. Slightly run-on sentences and parentheticals are fine; clean marketing prose makes the generator look better than it is.
- Cover at least one `fieldType`, edge case, or principle that existing cases don't.
- Prefer machine-checkable expectations (`slots.count`, `fields.field_types_subset_of`, regex patterns) over prose. A `groupBy_should_be_null` boolean is easier to assert than "the grouping should be reasonable".
- Include the regression context in `rationale` when the case targets a specific past bug.

## Regression targets (don't let these break)

- Cross-product expansion: 3×12 grid produces 36 slots, every `values` non-empty (`ev_021`).
- `groupBy` correctness: small named dimension → `groupBy=<that ref>`; pure date list → `null` (`ev_021`, `ev_022`, `ev_023`).
- Server forces `state='draft'` and writes `templateId='magic-compose'` in activity (`ev_018`).
- Empty prompt rejected with `invalid_input` before LLM call (`ev_014`).
- No PII fields, no tenant IDs in model output (`ev_012`, `ev_020`).
- Refusal pattern on hostile prompts (`ev_013`).
